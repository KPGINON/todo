# AI 重排考量逻辑（REORDER_LOGIC.md）

> 本文档定义"新增任务后自动重排"的排序逻辑，是 `lib/ai.js` 中 `suggestReorder` 的 prompt 依据，也是未来调整排序权重时的权威说明。
>
> 触发时机：每次新增任务后，前端携带 `newTodoId` 调用 `/ai/suggest-reorder`，AI 对**今天所有未完成任务**（含新增的那条）给出新顺序并直接应用，顶部提示条可撤销。任务 < 3 条时跳过，不调用 AI。

---

## 1. 设计目标

用户每加一条任务，系统就重新评估"今天该按什么顺序做"。重排不是把新任务无脑插到最前面，而是把全部未完成任务当作一个整体，按"今天先做哪个最合理"重排。

核心权衡：**紧迫性 > 重要性 > 依赖 > 精力匹配 > 聚类 > 积压**。前四项决定"今天能不能做完"，后两项决定"做起来顺不顺"。

---

## 2. 评估维度（按重要性从高到低）

### 2.1 截止时间紧迫度（最高优先级）
- 综合 `dueDate` + `dueTime` 判断。有截止且临近当前时间的最优先。
- **已过截止时间的任务立即排首位**——它已经迟了，必须最先处理或明确改期。
- 仅有日期无时间的，按当天 23:59 处理；仅有时间无日期的，按今天处理。
- 无截止的任务在本维度内不加分，靠后续维度排序。

### 2.2 优先级
- `high` > `medium` > `low`。
- 当截止紧迫度相近时，优先级决定先后。
- 优先级是用户主观标注的"重要性"，不等于"紧迫性"——所以排在截止之后。

### 2.3 依赖关系
- 若任务标题或备注暗示依赖（如"写文档"依赖"调研完成"、"发版"依赖"代码评审"），**被依赖的任务排前**。
- 依赖判断只基于文本线索，不跨任务记忆历史依赖。
- 无法判断依赖时跳过本维度，不臆测。

### 2.4 难度与精力匹配
- 高难度任务（开发 / 设计 / 写作 / 方案 / 复杂决策）安排靠前，对应一天中精力最好的时段。
- 低难度任务（回复 / 确认 / 整理 / 打卡 / 例行操作）可靠后，适合精力下降时做。
- 目的：避免把费脑的事拖到下午精力低谷，导致拖延或质量下降。

### 2.5 标签聚类
- 同标签任务尽量相邻，减少上下文切换（如连续处理"会议"类、"代码"类）。
- 仅在同紧迫度 + 同难度档位内做聚类，不为了聚类牺牲前面的维度。
- 标签是软信号，聚类是"锦上添花"不是主导因素。

### 2.6 积压时长
- `createdAt` 较早且仍未完成的任务适当提前，避免无限顺延（日清机制会让未完成任务每天顺延到今天，但顺序一直靠后会一直被跳过）。
- 仅作为**同档位内的微调**，不越过截止/优先级。

---

## 3. 新增任务插队规则

新任务不自动插队。仅当满足以下**任一**条件才排到前面：

1. 优先级 ≥ 当前列表中的最高优先级；
2. 有截止时间，且早于现有任务中最早的截止时间；
3. 标题或备注含"紧急 / 立即 / 马上 / ASAP"等强信号词。

否则按常规顺序追加（通常在默认排序的相应档位里，不会挤掉已有紧迫任务）。

这条规则是为了防止"随手加的小事"把手上真正紧急的事顶下去——新任务默认是"候选"，要证明自己紧迫才能插队。

---

## 4. 输入与输出契约

### 输入（`suggestReorder(newTodo, remainingTodos)`）
AI 收到的每个任务对象包含：`id`、`title`、`priority`、`dueDate`、`dueTime`、`tags`、`notes`、`isNew`（`是`/`否` 标记刚新增的那条）、`createdAt`，以及当前时间。

### 输出
```json
{
  "order": ["任务id按建议顺序排列"],
  "reasoning": "一句话说明重排理由，重点说明新任务是否插队及原因"
}
```
- `order` 必须包含传入的**所有**任务 id，不能多也不能少。
- `reasoning` 给用户看，要简洁（一句话），用于顶部提示条。

### 校验与兜底（`ai.js` 已实现）
- 后端对返回的 `order` 做校验：过滤掉不存在的 id，把遗漏的 id 追加到末尾。
- 解析失败 / AI 报错时，重排静默放弃（提示条隐藏），不影响新增任务本身。

---

## 5. 顺序如何持久化与渲染

- 应用顺序时调用 `POST /todos/reorder`，`store.reorderTodos(ids)` 给每个被排的任务写入自增的 `order` 字段（当前最大 order 之后递增）。
- 新任务创建时（`createTodo`）默认 `order = 现有最大 order + 1`，即排到末尾；随后触发的新增后重排会重新分配 order。
- GET `/` 渲染时：
  - 有 `order` 字段的任务按 `order` 升序排在前；
  - 无 `order` 字段的老任务（兼容旧数据）按默认规则（优先级 → 截止日期 → 截止时间 → 创建时间）追加在后；
  - 已完成统一沉底，组内保持上述顺序。
- 这样**手动拖拽 / AI 重排的顺序不会被默认排序覆盖**（修复了早期版本每次刷新重排丢失的问题）。
- 撤销：前端在应用前记录 DOM 中的旧顺序，点"撤销"则用旧顺序再调一次 `/todos/reorder` 并刷新。

---

## 6. 边界情况

| 场景 | 行为 |
|---|---|
| 未完成任务 < 3 条 | 跳过 AI 调用，不重排（`routes/todos.js` 前置过滤） |
| AI 未配置（缺 base_url / key / model） | `/ai/suggest-reorder` 返回 `ok:false`，前端静默隐藏提示条 |
| AI 返回的 order 缺失或多出 id | 后端过滤 + 补全，保证 id 集合完整 |
| AI 请求超时 / 报错 | 提示条隐藏，新增任务保留（用默认顺序） |
| 已完成任务 | 不参与重排（`suggestReorder` 只传 `!completed` 的任务） |
| 新任务本身无截止/低优先级 | 不插队，按常规追加；reasoning 会说明"新任务无需插队" |

## 7. 与"AI 智能重排"（`/ai-reschedule`）的区别

| | 新增后自动重排 | AI 智能重排 |
|---|---|---|
| 触发 | 新增任务后自动 | 用户在页面手动输入当日情况描述后触发 |
| 输入 | 全部未完成任务（无额外描述） | 剩余未完成任务 + 用户口述的当日情况 |
| 用途 | 常规插入新任务后的顺序维护 | 突发情况（临时会议、精力变化、紧急插入）下的大幅重排 |
| 函数 | `ai.suggestReorder` | `ai.reschedule` |

两者共用同一套考量维度，但 `reschedule` 额外把用户的"当日情况描述"作为最高优先信号叠加在 2.1 之上。

## 8. 截图排程（`ai.analyzeScreenshot`）

用户粘贴/拖入截图（日历、会议邀请、待办清单、聊天记录、邮件等），vision 模型识别其中的任务**并新增**到清单，再结合现有未完成任务整体重排。

### 与新增后重排的区别
- **新增后重排**：只对已有任务重排，不创建任务。
- **截图排程**：先从图片识别出新任务（`newTodos`），创建后纳入排程；输出 `order` 里用 `new-0`/`new-1`... 引用尚未持久化的新任务，后端创建后替换为真实 id 再 `reorderTodos`。

### 考量叠加规则
排程维度与第 2 节完全一致（截止紧迫度 > 优先级 > 依赖 > 难度精力 > 标签聚类 > 积压），但**截图识别出的截止时间/优先级信号优先**：
- 截图中明确的会议时间、deadline 作为高权重截止信号（等同 2.1）。
- 截图识别的"紧急/领导要求"等关键词提升优先级（等同 2.2）。
- 若截图带来多个同领域任务（如一场会议衍生 3 个事项），标签聚类（2.5）会让它们相邻。

### 识别规则
- 标题动宾结构、可执行、≤30 字；去口语化。
- 相对日期结合当前时间换算为 YYYY-MM-DD；时间转 24 小时制 HH:MM；未提及则留空。
- 优先级：紧急/马上/领导/重要 → high，有空/顺便/不急 → low，常规 → medium。
- 标签按领域 1-3 个；备注捕捉地点/参与人/依赖。
- 截图无任何可执行任务时 `newTodos` 返回空数组，仅对现有任务排程。

### 模型选择
优先用 `visionModel`（设置页配置的图像识别模型），留空时回退到 `aiModel`。请求用 OpenAI 兼容的 `image_url`（data URL base64）多模态格式，超时 60s（图像比文本慢）。

### 前端交互
今日页 🖼 按钮 → 浮层支持粘贴（Ctrl+V）/拖入/点击选择 → 预览 → "分析" → 后端**只分析不创建**，返回识别到的新任务（可编辑/删除）、完整排程顺序、reasoning → 用户确认后点"确认并排程" → `POST /ai/apply-schedule` 创建新任务 + 应用 reorder → 刷新页面。删除卡片时前端用 `data-orig-idx` 把 AI 原始 `new-N` 引用重映射到保留卡片的提交下标，保证 order 与实际创建任务对齐。识别为空时仍可"仅重排现有任务"。
